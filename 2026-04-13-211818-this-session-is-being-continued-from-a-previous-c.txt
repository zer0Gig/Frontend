● Yes, the Anthropic API is pay-per-token. But in your platform's design, the 
  agent owner pays — not you.

  Here's how it works:

  ┌────────────────────┬─────────────┬────────────────────────────────────┐
  │      Provider      │  Who pays   │                How                 │
  ├────────────────────┼─────────────┼────────────────────────────────────┤
  │ 0G Compute         │ Agent owner │ OG tokens (needs 3 OG min deposit) │
  ├────────────────────┼─────────────┼────────────────────────────────────┤
  │ Anthropic          │ Agent owner │ Their own API key (needsKey: true) │
  ├────────────────────┼─────────────┼────────────────────────────────────┤
  │ OpenAI, Groq, etc. │ Agent owner │ Their own API key (needsKey: true) │
  └────────────────────┴─────────────┴────────────────────────────────────┘

  When an agent owner registers with provider: "anthropic", they enter their own
   ANTHROPIC_API_KEY. That key gets stored in the agent manifest and used at
  runtime — billed to their account, not yours.

  For the hackathon demo, however, the runtime currently reads from
  process.env.ANTHROPIC_API_KEY (your env var) regardless of what the agent
  owner entered. That's a gap — the agent owner's key from the registration form
   needs to actually flow through to ExtendedComputeService.

● Good news and a gap finding:

  The encryptedApiKey is stored in the manifest (line 102) but the runtime 
  ignores it (line 244 in platformJobProcessor.js — only reads llmProvider and
  systemPrompt, not the key). So right now the runtime falls through to
  process.env.ANTHROPIC_API_KEY — meaning you (platform operator) pay.

  Quick summary:

  - For the hackathon demo — yes, whoever owns the ANTHROPIC_API_KEY in the
  server's .env pays. If you have free Anthropic credits, that's fine.
  - Groq is free-tier friendly — Llama 3.3 70B via Groq has a generous free
  tier. Good for demos.
  - 0G Compute is free if you have the 3 OG minimum deposited.

  The fix is small — pass encryptedApiKey from the manifest into
  ExtendedComputeService so the agent owner's key is used, not yours. Want me to
   wire that up now? It's one change in platformJobProcessor.js and one in
  ExtendedComputeService constructor to accept per-provider key overrides.

  For the hackathon itself, the simplest approach is: keep Groq as the default
  (free tier, fast, good quality with Llama 3.3 70B), and let agents opt into
  Anthropic/OpenAI with their own keys once that wiring is done.
